# 市场区卡牌交互升级方案（预览优先）

**日期**: 2026-04-28
**类型**: UI/交互提案（先立文档，后实施）
**目标**: 将市场区与 **牌库保留** 的点击行为统一为「先预览（大图/大卡背）→再动作执行」；**保留区最多 3 张牌的硬规则不变**。

## 1. 目标

将当前市场区行为从：

- 可买直接买卡（Single click）
- 不可买才打开预览
  改为：
- **任意左键点击市场卡先打开大图预览**
- 在大图内通过 **图下方两个主按钮** 分别执行 **买牌** 与 **保留**（买牌接法对齐保留区大图预览现有主按钮）
- **主按钮布局写死为「左买右保」**：在 LTR 界面下，Featured 卡图下方操作区从左到右依次为 **买牌**（左）、**保留**（右）；RTL 若需镜像，实现层对调物理位置但语义仍为「先买后保」阅读顺序由设计再定。
- **退役「先点金币再保留」**：不再支持「先在盘面选取金币（进入 `reserveGoldSelection` / `reserveModeActive`）→ 再单击市场卡即直接 `handleReserveCard`」的保留快捷路径；**从市场保留一张牌的合法发起方式仅保留一种——点击该市场卡打开预览 → 点右侧「保留」主按钮**（若规则仍要求指定金位，在按下「保留」之后的后续步骤里完成，不得绕过预览在卡面上直保留）。
- **熟练玩家快捷购买（范围决策）**：右键 / 拖拽直接购买纳入 **Phase 2**，**本轮不实施**；本轮仅实现左键预览优先主线，避免实现歧义。
- **按钮显隐硬规则（本轮）**：
    - 保留牌库牌（点牌堆）预览：**仅保留按钮**，且按钮 **居中**。
    - 购买已保留的牌（保留区卡牌预览）：**仅购买按钮**，且按钮 **居中**。
    - 选择皇室卡 / 查看自己或对手牌库：**两个按钮都不出现**。
    - 点击市场卡牌预览：**买牌+保留两个按钮必须同时出现**；买不起则买牌按钮禁用变灰；不能保留（如保留区满）则保留按钮禁用变灰。
- **预览交互一致性**：上述预览交互实现风格与状态切换应对齐「选皇室卡」预览（同层级反馈、同禁用态视觉与同关闭行为语义）。

这样可以减少手误，提升决策清晰度，且与当前保留区「预览后确认」模式一致。
**不变量**：玩家保留区 **最多容纳 3 张保留牌** 的硬限制维持现状；满 3 时市场/牌库的「保留」入口应按现有规则禁用或等价处理（本方案不改变该域规则，只改 UI 发起路径）。

## 2. 当前行为速览（已知基线）

- 市场卡点击入口在 `packages/ui/src/components/market/MarketLevelRow.tsx` 与 `packages/ui/src/components/Market.tsx`（当前：可买即直接触发购买）。
- **当前「先金后市场」保留路径（拟退役）**：`GamePlaySurface` 将 `reserveGoldSelection` 映射为 `Market` 的 `reserveModeActive`；为真时 `MarketLevelRow` 上 `reserveOnClick` 生效，且 `Market.handleBuy` 在点击卡面时改走 `handleReserveCard`——即用户先点盘面金币再点市场卡可一步保留。本方案要求 **关闭该通路**，使保留仅能从预览右钮发起。
- 市场卡预览目前由 `onPreviewCard` 回调驱动，`apps/desktop/src/app/shell/GameShell.tsx` 只负责把预览卡片传给 `CardPreviewOverlay`。
- 预览层 `packages/ui/src/components/CardPreviewOverlay.tsx` 已支持 `primaryActionLabel` + `onPrimaryAction`，可直接承担 **单侧** 主动作；双钮需扩展（见 §3.1）。

## 3. 明日实施内容（提案）

### 3.1 市场卡交互行为（核心）

1. 修改市场卡点击策略：
    - 不管是否可买，**左键**点击市场牌先打开大图预览。
    - 取消“可买卡直接执行 `initiateBuy`”路径（左键路径）。
2. **大图下方两个主按钮（左买右保）**
    - 与「单一主按钮、文案在买/保留之间切换」不同：预览打开后，Featured 卡图下方 **固定** 两个主操作位：**左侧 = 买牌**，**右侧 = 保留**（视觉权重同级；LTR 下物理顺序不得对调，除非 RTL 镜像策略单独成文）。
    - **左侧买牌**：行为与词条对齐保留区「保留牌大图预览」里已有的主按钮买牌路径——`packages/ui/src/components/playerZone/PlayerZoneReservedColumn.tsx` 中 `CardPreviewOverlay` 的 `primaryActionLabel`（`getLexiconLabel('buyCard', locale)`）+ `onPrimaryAction`（调用 `onBuyReserved`）这一套；市场侧应对齐 **同一模式**（同一 overlay API 或抽成共用的小函数/封装），即：可买时点击 **左钮** → 走 `initiateBuy(card, 'market', context)`（或现有等价入口）。
    - **右侧保留**：可保留且满足规则时启用，点击 **右钮** → `handleReserveCard(card, context)`（或现有等价入口）；若规则要求选金，则关闭预览并进入现有 `RESERVE_WAITING_GEM` / 选金流程；选金阶段可 Cancel 取消整次保留；若无可用 gold 则直接完成保留。
    - 若某一动作当前不可用（不可买 / 不可保留 / 非本方回合等）：对应按钮默认 **禁用，不隐藏**（与另一按钮独立），保证布局稳定。仅当两按钮均不可用且产品明确选择“纯详情模式”时，才允许隐藏整行按钮。
3. **实现备注**：当前 `CardPreviewOverlay` 仅暴露单个 `primaryActionLabel` / `onPrimaryAction`；要落地双主按钮需 **扩展组件**（例如增加 `secondaryActionLabel` + `onSecondaryAction`，或底部 `actions` 插槽）。扩展时 **约定主槽=左买、次槽=右保**（与上条「左买右保」一致）。保留区若暂时只需买牌一钮，可只接主槽、次槽留空。
4. **牌库卡背预览技术约束**：牌库预览不得依赖 `cards.length > 0` 才能打开。`CardPreviewOverlay` 需支持非 Card 正面内容（例如 `deckBackPreview` variant、`renderPreviewContent` 插槽、以及 actions 插槽）；禁止通过构造“假 card”绕过渲染分支。
5. **按钮触发语义（统一）**：先做预校验；预校验通过则 dispatch 对应动作并关闭预览；预校验失败则不 dispatch、不关闭，并展示错误。
6. 点击预览关闭（遮罩/X）不执行任何游戏动作。

### 3.1.1 各预览场景按钮矩阵（硬约束）

- **市场卡牌预览**：必须出现两个按钮（左买右保）；买不起 → 买牌禁用变灰；不能保留（如保留区满）→ 保留禁用变灰。
- **牌库保留预览（点牌堆）**：仅出现「保留」按钮，按钮居中。
- **已保留牌购买预览（保留区）**：仅出现「购买」按钮，按钮居中。
- **选择皇室卡预览**：不出现买/保留按钮。
- **查看自己或对手牌库预览**：不出现买/保留按钮。

### 3.2 流程图（文本）

```
用户左键点击市场卡
  → 打开 CardPreviewOverlay（single）
      → 图下方两枚主按钮（LTR）：[ 买牌（左）] [ 保留（右）] — 各自独立可用/禁用
          → 用户点左侧「买牌」且当前可买
              → 预校验通过后 initiateBuy（或等价）并关预览
          → 用户点右侧「保留」且当前可保留
              → 预校验通过后 handleReserveCard（或等价）并关预览
              → 需选金则进入 RESERVE_WAITING_GEM，支持 Cancel；无 gold 则直接完成保留
          → 用户关闭预览
              → 无动作
（不再存在：先点盘面金币 → 再点市场卡面即保留）
```

### 3.3 状态边界

- 非本方回合 / 回合末尾 / REVIEW / GAME_OVER：
    - 仍允许预览，动作按钮禁用（或不展示）。
- 在线对战（`ONLINE_MULTIPLAYER`）保持现有权限计算。
- 市场补位刷新动画 (`pendingMarketRefillSlots`) 不受影响。
- **保留区 3 张上限**：`playerReserved` 单侧已满 3 张时，市场预览 **右保**、牌库预览 **保留** 钮均须不可用（与当前规则一致，不在此方案中放宽）。
- **牌库保留（点牌堆）**：纳入本方案，行为见 **§3.5**（与「市场左买右保」区分：牌库预览 **仅保留单钮**，无购买）。

### 3.3.1 退役范围（与「左买右保」配套）

- **关闭**：`reserveModeActive` 驱动下，市场卡 **单击卡面即 `handleReserveCard`** 的行为（含 `Market.handleBuy` 在 `reserveModeActive` 时改道保留、`MarketLevelRow` 的 `reserveOnClick` 直保留）。
- **关闭**：依赖「先选取盘面金币使 `reserveGoldSelection` 为真，再点市场卡完成保留」的 UX；盘面金币选取若仍服务于其它规则（非市场保留），可保留或另行收敛，但 **不得再作为市场保留的发起条件**。
- **保留**：从市场 **保留** 的唯一发起入口 = **点市场卡 → 预览 → 右保**。

### 3.4 词条与可见性（建议）

- 使用已存在词条（不要硬编码中文）：
    - 买牌：`buyCard`
    - 保留：`card.reserve` / 当前保留按钮词条
    - 牌库预览标题：见 **§3.5 第 2 条**（lexicon 键名在实现 PR 中敲定）
- 保持大图中当前卡片的 `size="featured"`/`Card` 渲染不变；牌库分支使用 **卡背** 资产 + 同壳层布局参数。

### 3.5 牌库保留（点牌堆）：大卡背预览 + 单「保留」钮

与市场对翻 **正面** 不同，牌库侧展示 **牌背/卡背大图**（与当前等级牌背视觉一致，放大到与预览壳层匹配的 featured 尺度）。

1. **入口**：用户点击某等级 **牌库**（`handleReserveDeck` 对应 UI）→ 打开与 `CardPreviewOverlay` 同壳的预览（可为 `mode`/variant 区分「市场正面」「牌库卡背」）。
2. **标题栏**：提示语义为 **「保留牌库顶的牌」**（实现时用 lexicon：优先复用既有键；若无则 **新增词条键** 并在中英目录补译，**禁止**在组件内硬编码中文字符串）。
3. **底部主操作区**：**仅一枚主按钮——「保留」**，并 **居中摆放**；**不展示购买按钮**（与市场的「左买右保」刻意区分）。按钮不可用条件与满 3 张、非己方回合等现有规则对齐。
4. **确认保留之后**（与现行逻辑一致）：
    - 进入 **选取盘面金币**（`reserveGoldSelection` 等）环节，用于指定支付金位；
    - 在该选金环节中，用户仍可通过 **Cancel / 取消保留** 整段撤销（与 **现在** 行为一致，不在此方案中削弱）。
5. **无金可点**：若当前盘面 **不存在可作为保留代价的金币格**（或等价判定：无可用 gold 位），则用户在预览中点「保留」后 **直接进入完成保留**（跳过选金子流程），避免死锁。
6. 预览关闭（遮罩/X）在未确认保留前：**不提交**保留、不进入选金（与市场预览规则一致）。

### 3.6 熟练玩家快捷购买（本期范围）

- **范围结论**：右键 / 拖拽直接购买列入 **Phase 2**，本轮不实施。
- 本轮统一口径：左键走预览优先主线；右键与拖拽不承担直接购买。
- Phase 2 预留规则（后续评审时启用）：
    - 右键点击可买市场卡：可直接购买。
    - 拖动可买市场卡到己方牌区：可直接购买。
    - 右键 / 拖拽仅代表购买，不代表保留。
    - 不可买、非本方回合、`REVIEW`、`GAME_OVER`、pending refill 时不得触发。

## 5. 验收标准（建议）

- 左键点击市场任意卡都打开大图（本轮）；右键/拖拽快捷购买为 Phase 2 保留项。
- 预览内 **同时展示** 买牌 / 保留两个主级入口，且 **LTR 下固定左买右保**（各自独立可用或禁用）；可买卡必须经过 **左侧** 买牌按钮才能购买。
- 市场预览下，买/保两个按钮 **必须同时出现**；买不起仅买牌禁用变灰；保留位满仅保留禁用变灰。
- 可保留时 **右侧** 保留按钮行为正确；买牌按钮与保留区大图买牌在交互与词条上一致。
- **从市场发起保留** 仅能通过「点市场卡 → 预览 → 右侧保留」；**不存在**「先点盘面金币再点市场卡即完成保留」。
- 市场右保与牌库保留在选金规则上一致：需选金则进入 `RESERVE_WAITING_GEM` 且可 Cancel；无 gold 则直接完成保留。
- 预览关闭不会触发动作。
- 现有保留区（`PlayerZone`）**三张保留位硬上限** 与满位禁用行为与现网一致。
- **牌库保留**：先大卡背预览 + 标题提示 + **仅保留按钮**；确认后进入选金；选金阶段 **Cancel 仍可取消整次保留**；**场上无金** 时确认后 **直接保留** 无选金子流程。
- **已保留牌购买**：预览中仅出现购买按钮且居中。
- **选皇室卡 / 查看自己或对手牌库**：预览不出现买/保按钮。
- 不引入新的文案耦合（继续使用 lexicon；牌库标题若新增键则一并纳入 lexicon 与回归用例）。

## 6. 预期收益与风险

- 预期收益：
    - 决策清晰度提升；误点买牌率下降。
    - 与保留区“先看大图再确认”交互一致。
- 主要风险：
    - 玩家回合耗时增加（从1步买牌到2步买牌）。
    - 习惯「先点金再点市场卡」的玩家需要 **发版说明 / 内测提示** 引导至预览右保；牌库保留从「一点即进选金」变为「先预览再保留」亦需一句说明。
    - 两主按钮并列后，一般不再做「整颗主按钮在买/保留之间切换」；若业务仍要求保留意图强提示，可通过右钮高亮或短提示条解决，而非恢复金后市场直保留。
- 回滚策略：
    - 保留当前代码为灰度备份逻辑；必要时恢复「可买直接买」路径。
